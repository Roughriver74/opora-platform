## 1. Обзор продукта

### 1.1 Vision

OPORA — это B2B SaaS-платформа для компаний, управляющих полевыми командами. Продукт позволяет регистрировать, отслеживать и анализировать визиты представителей к клиентам, синхронизировать данные с CRM-системами и предоставлять руководству прозрачную картину работы команды в режиме реального времени.

### 1.2 Цель продукта

| Для кого | Что даёт OPORA |
|----------|----------------|
| Руководитель | Прозрачность работы полевой команды, аналитика визитов, KPI |
| Полевой представитель | Быстрая мобильная регистрация визита, история контактов |
| IT / CRM-администратор | Готовая интеграция с Bitrix24, гибкие поля через JSONB |
| Платформенный администратор | Управление тарифами, организациями, мониторинг |

### 1.3 Ключевые метрики успеха (KPI)

| Метрика | Целевое значение | Период |
|---------|-----------------|--------|
| MRR (Monthly Recurring Revenue) | 150 000 ₽ | 12 мес |
| Количество платящих организаций | 30+ | 12 мес |
| Retention Rate (платные) | ≥ 85% | ежемесячно |
| Median визитов в день на платформе | ≥ 500 | 6 мес |
| Uptime | ≥ 99.5% | ежемесячно |
| P95 latency API | < 300 мс | постоянно |
| TTFB (Time to First Byte, frontend) | < 1.5 с | постоянно |
| Customer Support tickets / org / мес | < 2 | 6 мес |

## 2. Целевая аудитория

### 2.1 Персоны пользователей

#### Персона 1 — Региональный менеджер (org_admin)

- **Возраст:** 35–50 лет
- **Отрасль:** Фармацевтика, FMCG, медицинское оборудование
- **Контекст:** Управляет командой 5–20 полевых представителей. Работает в офисе, использует десктоп.
- **Боли:**
  - Не понимает, кто и когда посещал клиентов
  - Отчёты в Excel устарели к моменту получения
  - Данные разрознены: CRM + мессенджеры + звонки
- **Jobs to be done:**
  - *Когда* я хочу оценить эффективность команды, *я хочу* видеть дашборд с визитами по сотрудникам и клиентам, *чтобы* принимать кадровые и коммерческие решения.
  - *Когда* я настраиваю новый процесс, *я хочу* задать произвольные поля для визитов, *чтобы* собирать нужные данные без участия разработчика.

#### Персона 2 — Медицинский/торговый представитель (user)

- **Возраст:** 25–40 лет
- **Контекст:** Проводит 5–15 визитов в день. Работает со смартфона в дороге.
- **Боли:**
  - Долго заполнять отчёт после визита
  - Нет мобильного приложения — форма неудобна на телефоне
  - Забывает детали контакта через несколько часов
- **Jobs to be done:**
  - *Когда* я только что провёл встречу, *я хочу* зафиксировать визит за 30 секунд, *чтобы* не тратить вечер на отчёты.
  - *Когда* я еду к клиенту, *я хочу* быстро найти историю прошлых визитов к нему, *чтобы* подготовиться к встрече.

#### Персона 3 — Платформенный администратор (platform_admin)

- **Контекст:** Управляет всей мультитенантной платформой. Это внутренняя роль OPORA.
- **Jobs to be done:**
  - Подключать новые организации, управлять тарифами и лимитами.
  - Видеть агрегированную статистику по платформе, выявлять проблемные тенанты.
  - Имперсонировать org_admin для диагностики проблем.

### 3.2 Выявленные проблемы

#### Критические (блокируют масштабирование)

| Проблема | Область | Описание |
|----------|---------|----------|
| Нет rate limiting | Безопасность | API открыт для brute-force атак |
| Нет refresh tokens | Безопасность | Сессии не продлеваются безопасно |
| Нет security headers | Безопасность | Отсутствуют CSP, X-Frame-Options и др. |
| Нет валидации паролей | Безопасность | Принимаются пароли любой сложности |
| N+1 запросы к БД | Производительность | Неоптимальные ORM-запросы |
| Нет индексов на org_id | Производительность | Полное сканирование таблиц при фильтрации |
| Нет connection pooling | Производительность | Пул соединений не настроен |
| Bundle 1.1 MB без code splitting | Frontend | Медленная загрузка на мобильных |
| Компоненты 3000+ строк | Frontend | Сложно поддерживать и тестировать |
| 0 тестов | Качество | Невозможно безопасно рефакторить |
| Нет CI/CD | DevOps | Ручной деплой, высокий риск ошибок |

## 5. Функциональные требования

### 5.1 Фаза 1 — Критические фиксы безопасности и производительности (1–2 недели)

---

#### FR-1.1 Rate Limiting

**User Story:** Как платформенный администратор, я хочу, чтобы API был защищён от перебора паролей и DDoS, чтобы платформа оставалась доступной для легитимных пользователей.

**Acceptance Criteria:**
- [ ] Endpoint `POST /api/auth/login` ограничен до 5 запросов/мин с одного IP
- [ ] Endpoint `POST /api/auth/register` ограничен до 3 запросов/мин с одного IP
- [ ] Все остальные API endpoints ограничены до 100 запросов/мин на пользователя
- [ ] При превышении лимита возвращается HTTP 429 с заголовком `Retry-After`
- [ ] Лимиты конфигурируются через переменные окружения

---

#### FR-1.2 Security Headers

**User Story:** Как пользователь, я хочу, чтобы платформа была защищена от XSS, clickjacking и других веб-уязвимостей.

**Acceptance Criteria:**
- [ ] Все ответы API содержат заголовки: `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `X-XSS-Protection: 1; mode=block`
- [ ] Настроен `Content-Security-Policy` для frontend (через Nginx)
- [ ] Настроен `Strict-Transport-Security` для production (HTTPS)
- [ ] Заголовок `Server` не раскрывает информацию о стеке

---

#### FR-1.3 Валидация паролей

**User Story:** Как org_admin, я хочу, чтобы платформа требовала надёжные пароли, чтобы аккаунты организации были защищены.

**Acceptance Criteria:**
- [ ] Пароль должен содержать минимум 8 символов
- [ ] Пароль должен содержать минимум одну цифру и одну букву
- [ ] При несоответствии требованиям API возвращает HTTP 422 с понятным описанием ошибки
- [ ] Frontend отображает требования к паролю в реальном времени при вводе
- [ ] Требования к паролю одинаковы при регистрации и смене пароля

---

#### FR-1.4 Индексы базы данных

**User Story:** Как пользователь, я хочу, чтобы списки визитов и компаний загружались быстро, даже при большом объёме данных.

**Acceptance Criteria:**
- [ ] Созданы индексы: `visits(org_id)`, `visits(org_id, date)`, `companies(org_id)`, `contacts(org_id)`, `doctors(org_id)`
- [ ] Применена Alembic-миграция с добавлением индексов
- [ ] Time запроса списка визитов за месяц (1000+ записей) < 100 мс (EXPLAIN ANALYZE)
- [ ] Индексы покрывают все основные фильтры (org_id + date range, org_id + user_id)

---

#### FR-1.5 Connection Pooling

**User Story:** Как DevOps, я хочу, чтобы backend не исчерпывал соединения с БД под нагрузкой.

**Acceptance Criteria:**
- [ ] Настроен SQLAlchemy async connection pool: `pool_size=10`, `max_overflow=20`
- [ ] Таймаут получения соединения из пула: `pool_timeout=30`
- [ ] Параметры пула конфигурируются через переменные окружения
- [ ] В логах отображается статус пула при старте приложения

---

### 5.2 Фаза 2 — Надёжность и производительность (2–4 недели)

---

#### FR-2.1 JWT Refresh Tokens

**User Story:** Как пользователь, я хочу оставаться авторизованным в течение рабочего дня без повторного ввода пароля, но с автоматическим выходом при длительном бездействии.

**Acceptance Criteria:**
- [ ] Access token: TTL 15 минут
- [ ] Refresh token: TTL 7 дней, хранится в HttpOnly cookie
- [ ] Endpoint `POST /api/auth/refresh` принимает refresh token и возвращает новый access token
- [ ] Endpoint `POST /api/auth/logout` инвалидирует refresh token (хранение в Redis или БД)
- [ ] При истечении access token frontend автоматически запрашивает новый через refresh endpoint
- [ ] Повторное использование отозванного refresh token возвращает HTTP 401 и инвалидирует все сессии пользователя (rotation)

---

#### FR-2.2 Structured Logging

**User Story:** Как DevOps, я хочу видеть структурированные логи с контекстом запроса, чтобы быстро диагностировать проблемы в production.

**Acceptance Criteria:**
- [ ] Все логи в формате JSON (structlog или loguru)
- [ ] Каждый лог-запись содержит: `timestamp`, `level`, `request_id`, `user_id`, `org_id`, `method`, `path`, `status_code`, `duration_ms`
- [ ] Ошибки 500 логируются с полным stack trace
- [ ] Логи доступны через `docker-compose logs -f backend`
- [ ] Log level конфигурируется через `LOG_LEVEL` переменную окружения

---

#### FR-2.3 Устранение N+1 запросов

**User Story:** Как пользователь, я хочу, чтобы список визитов с информацией о компании и пользователе загружался за один запрос к API.

**Acceptance Criteria:**
- [ ] Endpoint `GET /api/visits` выполняет не более 3 SQL-запросов при любом объёме данных (проверяется через `sqlalchemy.event.listen`)
- [ ] Использованы `selectinload` или `joinedload` для связанных сущностей (User, Company) при загрузке списка визитов
- [ ] Аналогично для списка компаний с адресами (`GET /api/clinics`)
- [ ] Regression-тест: запрос 100 визитов < 200 мс (локальная БД)

---

#### FR-2.4 Redis-кэширование

**User Story:** Как пользователь, я хочу, чтобы статичные справочники (настройки, маппинги полей) загружались мгновенно.

**Acceptance Criteria:**
- [ ] Redis подключён как зависимость в docker-compose
- [ ] `GET /api/settings` кэшируется в Redis на 5 минут с инвалидацией при `PUT /api/settings`
- [ ] `GET /api/admin/field-mappings` кэшируется на 10 минут
- [ ] При недоступности Redis приложение продолжает работу (fallback без кэша, логирование ошибки)
- [ ] Время отклика закэшированных endpoint < 20 мс

---

#### FR-2.5 Code Splitting (Frontend)

**User Story:** Как пользователь мобильного браузера, я хочу, чтобы первый экран загружался быстро, даже при медленном интернете.

**Acceptance Criteria:**
- [ ] Начальный bundle (main chunk) < 300 KB gzip
- [ ] Lazy loading для роутов: Admin panel, Settings, Platform Admin загружаются по требованию
- [ ] `React.lazy()` + `Suspense` с Skeleton-плейсхолдером для каждого lazy-роута
- [ ] Lighthouse Performance score > 75 на мобильном устройстве (throttled 4G)
- [ ] Компоненты > 500 строк разбиты на subcomponents с отдельными файлами

---

### 5.3 Фаза 3 — Продуктовые фичи (4–8 недель)

---

#### FR-3.1 Analytics Dashboard

**User Story:** Как org_admin, я хочу видеть сводную аналитику по работе команды, чтобы принимать управленческие решения на основе данных.

**Acceptance Criteria:**
- [ ] Дашборд доступен по роуту `/dashboard` для ролей org_admin и user (с фильтром по себе)
- [ ] Метрики: количество визитов за период, по сотрудникам, по типам, по статусам
- [ ] График визитов по дням (за выбранный период, по умолчанию — текущий месяц)
- [ ] Топ-10 компаний по количеству визитов
- [ ] Фильтры: период (произвольный диапазон дат), сотрудник, тип визита
- [ ] Данные агрегируются на backend (не на frontend) с использованием SQL GROUP BY
- [ ] Время загрузки дашборда < 2 с при 10 000 визитов в базе

---

#### FR-3.2 Принудительное ограничение тарифов (Tiered Pricing Enforcement)

**User Story:** Как платформенный администратор, я хочу, чтобы организации на тарифе Free не превышали лимиты, чтобы платные функции оставались стимулом для апгрейда.

**Acceptance Criteria:**
- [ ] При создании визита проверяется лимит `plan_limits.visits_per_month` из Organization
- [ ] При превышении лимита API возвращает HTTP 402 с сообщением "Достигнут лимит тарифного плана"
- [ ] При попытке добавить пользователя сверх `plan_limits.max_users` — HTTP 402
- [ ] Frontend отображает баннер "X из Y визитов использовано" для Free-тарифа
- [ ] При апгрейде тарифа лимиты обновляются немедленно (без перезапуска)
- [ ] platform_admin может изменить лимиты организации через UI

---

#### FR-3.3 Аудит-лог

**User Story:** Как org_admin, я хочу видеть историю изменений в системе, чтобы понимать, кто и что изменил.

**Acceptance Criteria:**
- [ ] Создана модель `AuditLog`: `id`, `org_id`, `user_id`, `action` (enum), `entity_type`, `entity_id`, `changes` (JSONB), `created_at`
- [ ] Логируются события: создание/изменение/удаление визита, создание/удаление пользователя, изменение настроек
- [ ] Endpoint `GET /api/audit-log` доступен только для org_admin, с пагинацией и фильтрами по типу события и периоду
- [ ] Аудит-записи неизменяемы (нет PUT/DELETE)
- [ ] Хранение аудит-лога: 90 дней для Free, 365 дней для Paid

---

#### FR-3.4 PWA (Progressive Web App)

**User Story:** Как представитель, я хочу устанавливать приложение на экран смартфона и пользоваться им при слабом сигнале.

**Acceptance Criteria:**
- [ ] Добавлен `manifest.json` с иконками (192x192, 512x512) и `theme_color`
- [ ] Service Worker кэширует статические ассеты (CSS, JS, иконки) при первом посещении
- [ ] При оффлайне пользователь видит экран "Нет подключения" с кнопкой повтора вместо стандартной ошибки браузера
- [ ] Форма создания визита доступна офлайн (данные сохраняются в IndexedDB)
- [ ] При восстановлении соединения офлайн-визиты синхронизируются автоматически
- [ ] Lighthouse PWA score ≥ 90

### 5.4 Фаза 4 — Зрелость и масштабирование (8+ недель)

---

#### FR-4.1 CI/CD Pipeline

**User Story:** Как разработчик, я хочу, чтобы каждый пуш в main автоматически проходил проверки и деплоился, чтобы снизить риск ошибок при релизе.

**Acceptance Criteria:**
- [ ] GitHub Actions workflow: lint → tests → build → deploy при пуше в `main`
- [ ] Backend: `ruff` (linting), `mypy` (typing), `pytest` (unit + integration)
- [ ] Frontend: `eslint`, `tsc --noEmit`, `vitest` (unit), Playwright (e2e)
- [ ] Деплой происходит только при успешном прохождении всех проверок
- [ ] Среднее время pipeline < 10 минут
- [ ] Уведомления об ошибках pipeline в Telegram/Email

---

#### FR-4.2 Blue-Green Deployment

**User Story:** Как DevOps, я хочу выполнять деплой без downtime, чтобы пользователи не теряли доступ во время обновлений.

**Acceptance Criteria:**
- [ ] Настроен blue-green или rolling деплой через Dokploy
- [ ] Health check endpoint `GET /health` возвращает 200 при готовности сервиса
- [ ] Nginx переключает трафик на новый контейнер только после успешного health check
- [ ] Rollback к предыдущей версии занимает < 2 минут
- [ ] Downtime при деплое = 0 (для пользователей с активными сессиями)

---

#### FR-4.3 Мониторинг (Prometheus + Grafana)

**User Story:** Как DevOps, я хочу видеть метрики производительности и ошибок в реальном времени, чтобы реагировать на инциденты проактивно.

**Acceptance Criteria:**
- [ ] Exposed `/metrics` endpoint в Prometheus-формате (через `prometheus-fastapi-instrumentator`)
- [ ] Метрики: HTTP request rate, error rate (4xx/5xx), P50/P95/P99 latency, DB pool utilization, active users
- [ ] Grafana Dashboard с ключевыми метриками платформы
- [ ] Алерты: error rate > 5% за 5 мин, P95 latency > 1 с за 5 мин, DB connections > 80%
- [ ] Алерты доставляются в Telegram

---

#### FR-4.4 Тестовое покрытие

**User Story:** Как разработчик, я хочу иметь тестовое покрытие ключевых сценариев, чтобы безопасно вносить изменения.

**Acceptance Criteria:**
- [ ] Backend coverage ≥ 70% (pytest-cov)
- [ ] Покрыты unit-тестами: auth (login, refresh, logout), visit CRUD, тарифные лимиты, аудит-лог
- [ ] Покрыты integration-тестами: полный flow создания визита, импорт Excel, Bitrix24 webhook
- [ ] Frontend coverage ≥ 60% (vitest)
- [ ] E2E тесты (Playwright): login → create visit → logout; admin → create org → set plan
- [ ] Тесты запускаются в CI при каждом PR

## 7. Ограничения и зависимости

### 7.1 Технические ограничения

| Ограничение | Описание |
|-------------|---------|
| Single-region deployment | Текущий деплой на одном VPS через Dokploy; горизонтальное масштабирование ограничено |
| Bitrix24 API rate limits | Webhook ограничен частотой запросов со стороны Bitrix24 |
| YuKassa | Только рублёвые платежи, поддержка РФ |
| PostgreSQL JSONB | Динамические поля не индексируются автоматически; полнотекстовый поиск по JSONB требует GIN-индексов |

### 7.2 Внешние зависимости

| Сервис | Назначение | Критичность | Fallback |
|--------|-----------|-------------|---------|
| Bitrix24 | CRM синхронизация | Низкая | Работает без интеграции |
| DaData | Адресный автокомплит | Низкая | Ручной ввод адреса |
| YuKassa | Платежи | Средняя | Ручное управление тарифами |
| SMTP (внешний) | Уведомления | Низкая | Работает без уведомлений |
| Dokploy | Деплой и оркестрация | Высокая | Ручной docker-compose deploy |

### 7.3 Организационные ограничения

- Команда разработки: 1–2 разработчика
- Бюджет инфраструктуры: VPS с 4 vCPU / 8 GB RAM
- Соответствие законодательству: 152-ФЗ "О персональных данных" (данные хранятся на серверах в РФ)
