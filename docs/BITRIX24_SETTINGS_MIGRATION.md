# Миграция настроек Bitrix24 в базу данных

## Обзор

Система управления настройками Bitrix24 позволяет администраторам управлять интеграцией через веб-интерфейс без необходимости доступа к серверу и редактирования файлов конфигурации.

### Что изменилось

**До:**
- Настройки хранились в `.env` файлах
- Для изменения требовался доступ к серверу
- Перезапуск сервера при каждом изменении
- Невозможность управления через UI

**После:**
- Настройки в базе данных PostgreSQL
- Управление через админ-панель (Настройки → Bitrix24 Интеграция)
- Горячая перезагрузка без рестарта сервера
- Безопасное хранение с шифрованием AES-256-CBC

## Приоритет настроек

Система использует каскадный подход с приоритетом базы данных:

```
База данных (БД) > Environment переменные (.env)
```

### Как это работает

1. **При запуске сервера:**
   - Система проверяет наличие настроек `bitrix24.enabled` в БД
   - Если настройка найдена → используется значение из БД (даже если `false`)
   - Если настройки нет → fallback на `process.env.BITRIX24_ENABLED`

2. **При изменении настроек:**
   - Изменения сохраняются в БД
   - Кеш конфигурации инвалидируется
   - При следующем обращении читается обновленное значение

3. **Кеширование:**
   - In-memory кеш с TTL 60 секунд
   - Автоматическая инвалидация при изменениях
   - Снижает нагрузку на БД

## Безопасность

### Шифрование Webhook URL

Webhook URL содержит конфиденциальные данные и шифруется с использованием **AES-256-CBC**.

**Параметры шифрования:**
- Алгоритм: `aes-256-cbc`
- Ключ: `SHA256(JWT_SECRET)`
- IV: 16 случайных байтов (генерируется для каждого значения)
- Формат: `<IV в hex>:<зашифрованные данные в hex>`

**Процесс:**
```typescript
// Автоматическое шифрование при сохранении
settingsService.upsertSetting('bitrix24.webhook_url', 'https://...', {
  isEncrypted: true  // Триггер автошифрования
})

// Автоматическое дешифрование при чтении
const url = await settingsService.getSettingValueDecrypted('bitrix24.webhook_url')
```

### Безопасность в API

- Webhook URL в ответах API всегда маскируется: `https://***hidden***`
- Полные данные доступны только при дешифровании на backend
- Требуется JWT_SECRET для расшифровки (хранится только на сервере)

## Использование

### Через веб-интерфейс (UI)

1. **Открыть настройки:**
   - Войти как администратор
   - Перейти в раздел "Настройки системы"
   - Выбрать категорию "Bitrix24 Интеграция"

2. **Управление настройками:**
   - **Переключатель "Включить интеграцию"**: Включает/отключает Bitrix24
   - **Поле "Webhook URL"**: Адрес вебхука Bitrix24 REST API
   - **Кнопка "Тестировать подключение"**: Проверка корректности URL
   - **Кнопка "Сохранить"**: Сохранение настроек в БД
   - **Кнопка "Перезагрузить интеграцию"**: Применение изменений без рестарта

3. **Статусы:**
   - 🟢 Зеленый: Интеграция активна и работает
   - 🟡 Желтый: Настройки изменены, требуется перезагрузка
   - 🔴 Красный: Ошибка подключения или конфигурации

### Через API

#### Получить текущую конфигурацию
```http
GET /api/settings/bitrix24/config
```

**Ответ:**
```json
{
  "success": true,
  "data": {
    "enabled": true,
    "hasWebhookUrl": true,
    "webhookUrl": "https://your...***hidden***"
  }
}
```

#### Тестировать подключение
```http
POST /api/settings/bitrix24/test-connection
```

**Ответ (успех):**
```json
{
  "success": true,
  "message": "Подключение к Bitrix24 успешно",
  "data": {
    "fieldsCount": 145
  }
}
```

**Ответ (ошибка):**
```json
{
  "success": false,
  "message": "Ошибка подключения к Bitrix24: Timeout"
}
```

#### Перезагрузить интеграцию (только admin)
```http
POST /api/settings/bitrix24/reload
Authorization: Bearer <admin_jwt_token>
```

**Ответ:**
```json
{
  "success": true,
  "message": "Интеграция Bitrix24 успешно перезагружена"
}
```

#### Обновить настройки напрямую
```http
PUT /api/settings/bitrix24.enabled
Content-Type: application/json

{
  "value": true
}
```

```http
PUT /api/settings/bitrix24.webhook_url
Content-Type: application/json

{
  "value": "https://your-bitrix.bitrix24.ru/rest/1/xxxxx/"
}
```

## Миграция из .env в БД

### Автоматическая миграция

При запуске сервера автоматически выполняется TypeORM миграция:

```bash
docker compose exec backend npm run migration:run
```

**Что делает миграция:**
1. Проверяет существование настроек `bitrix24.enabled` и `bitrix24.webhook_url`
2. Если настроек нет → читает значения из `.env`
3. Создает записи в таблице `settings` с шифрованием webhook URL
4. **НЕ перезаписывает** существующие настройки в БД

### Ручная миграция

Для явного переноса настроек из `.env`:

```bash
docker compose exec backend npm run migrate:env-to-db
```

**Скрипт выполняет:**
1. Чтение `BITRIX24_ENABLED` и `BITRIX24_WEBHOOK_URL` из `.env`
2. Создание/обновление настроек в БД с автошифрованием
3. Проверка корректности миграции
4. Вывод детального отчета

## Откат и Troubleshooting

### Откат миграции

Если необходимо вернуться к использованию `.env`:

**Вариант 1: Удалить настройки из БД**
```sql
DELETE FROM settings WHERE key = 'bitrix24.enabled';
DELETE FROM settings WHERE key = 'bitrix24.webhook_url';
```

**Вариант 2: Откат через TypeORM**
```bash
docker compose exec backend npm run migration:revert
```

После удаления настроек из БД система автоматически вернется к чтению из `.env`.

### Распространенные проблемы

#### Проблема: "Ошибка дешифрования настройки"

**Причина:** Изменился `JWT_SECRET` после шифрования.

**Решение:**
1. Восстановить оригинальный `JWT_SECRET`
2. ИЛИ удалить зашифрованную настройку:
```sql
DELETE FROM settings WHERE key = 'bitrix24.webhook_url';
```
3. Ввести webhook URL заново через UI

#### Проблема: "Интеграция включена, но WEBHOOK_URL не установлен"

**Причина:** Настройка `bitrix24.enabled = true`, но URL отсутствует.

**Решение:**
- Открыть UI → Bitrix24 Интеграция
- Ввести корректный webhook URL
- Сохранить и перезагрузить

#### Проблема: Изменения не применяются

**Причины:**
1. Не нажата кнопка "Перезагрузить интеграцию"
2. Кеш не инвалидирован

**Решение:**
- Нажать "Перезагрузить интеграцию" в UI
- ИЛИ перезапустить сервер:
```bash
docker compose restart backend
```

#### Проблема: "Тест подключения" возвращает ошибку

**Причины:**
1. Неверный формат webhook URL
2. Недоступен Bitrix24 сервер
3. Некорректные права доступа для webhook

**Решение:**
1. Проверить формат URL (должен содержать `/rest/`)
2. Проверить доступность Bitrix24
3. Создать новый входящий webhook в Bitrix24 с правами на чтение сделок

### Валидация webhook URL

Система автоматически валидирует формат webhook URL:

**Допустимые форматы:**
```
https://ваш-домен.bitrix24.ru/rest/1/xxxxx/
https://ваш-домен.bitrix24.com/rest/1/xxxxx/
https://ваш-домен.bitrix24.net/rest/1/xxxxx/
https://ваш-домен.bitrix24.by/rest/1/xxxxx/
https://ваш-домен.bitrix24.kz/rest/1/xxxxx/
https://ваш-домен.bitrix24.ua/rest/1/xxxxx/
```

**Регулярное выражение:**
```
^https?://.*bitrix24\.(ru|com|net|by|kz|ua)/rest/
```

- ✅ **ВСЕГДА** используйте сильный `JWT_SECRET` (минимум 32 символа)
- ✅ Храните `JWT_SECRET` в защищенном хранилище (не в git)
- ✅ Регулярно обновляйте webhook URL при компрометации
- ✅ Используйте HTTPS для всех запросов к Bitrix24

## FAQ

**Q: Что произойдет, если удалить настройки из БД?**
A: Система автоматически перейдет в режим fallback и будет читать настройки из `.env`.

**Q: Нужно ли удалять настройки из .env после миграции?**
A: Не обязательно. Оставьте их как резервный fallback на случай проблем с БД.

**Q: Можно ли использовать разные настройки для разных окружений (dev/prod)?**
A: Да, используйте разные базы данных для каждого окружения.

**Q: Как восстановить доступ при утере JWT_SECRET?**
A: Необходимо удалить зашифрованную настройку `bitrix24.webhook_url` из БД и ввести ее заново.

**Q: Безопасно ли хранить webhook URL в БД?**
A: Да, при условии использования шифрования AES-256-CBC и защиты JWT_SECRET.

**Q: Можно ли откатиться к .env без потери данных?**
A: Да, просто удалите настройки из БД. Данные останутся в `.env`.
