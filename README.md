# Система учёта визитов (West Visit)

Прогрессивное веб-приложение (PWA) для учёта и управления визитами медицинских представителей компании. Интегрировано с Bitrix24 для двусторонней синхронизации данных о визитах, клиниках и врачах.

## Ключевые возможности

- **Визиты**: Полное управление визитами медицинских представителей с цветовой индикацией статусов
- **Клиники**: Детальный учёт клиник с возможностью синхронизации с Bitrix24
- **Врачи**: Управление информацией о врачах и их связями с клиниками
- **Интеграция с Bitrix24**: Двусторонняя синхронизация всех данных
- **Динамические поля**: Настраиваемые поля с поддержкой разных типов (текст, дата, время, списки и т.д.)
- **Отзывчивый интерфейс**: Адаптивный дизайн для комфортной работы на любых устройствах
- **Темная тема**: Современный UI с поддержкой темной темы на базе Material-UI
- **Оффлайн-режим**: Возможность работы без интернета с последующей синхронизацией

## Быстрый старт

### Требования

- Docker и Docker Compose
- Node.js 16+ (для локальной разработки)
- Python 3.11+ (для локальной разработки)

### Запуск для разработки (рекомендуемый способ)

1. Запустите базу данных:

```bash
./start-dev-db.sh
```

2. Запустите бэкенд:

```bash
./start-dev-backend.sh
```

3. Запустите фронтенд:

```bash
./start-dev-frontend.sh
```

Приложение будет доступно:

- Frontend: http://localhost:3000
- Backend API: http://localhost:8000
- API Документация: http://localhost:8000/docs

## Модели данных

### User (Пользователь)

- **id**: Уникальный идентификатор
- **email**: Email пользователя (совпадает с email в Bitrix24)
- **hashed_password**: Хешированный пароль
- **bitrix_user_id**: ID пользователя в Bitrix24
- **is_active**: Статус активности
- **is_admin**: Права администратора

### Company (Клиника)

- **id**: Уникальный идентификатор
- **bitrix_id**: ID компании в Bitrix24
- **name**: Название клиники
- **dynamic_fields**: JSON с динамическими полями из Bitrix24
- **last_synced**: Время последней синхронизации
- **sync_status**: Статус синхронизации

### Visit (Визит)

- **id**: Уникальный идентификатор
- **bitrix_id**: ID визита в Bitrix24
- **company_id**: Ссылка на клинику
- **user_id**: Ссылка на пользователя
- **date**: Дата/время визита
- **dynamic_fields**: JSON с динамическими полями из Bitrix24
- **status**: Статус визита (planned, success, fail)
- **visit_type**: Тип визита
- **comment**: Комментарий
- **with_distributor**: Флаг визита с дистрибьютором
- **sansus**: Дополнительный флаг
- **last_synced**: Время последней синхронизации
- **sync_status**: Статус синхронизации

### Doctor (ЛПР)

- **id**: Уникальный идентификатор
- **bitrix_id**: ID врача в Bitrix24
- **name**: ФИО врача
- **dynamic_fields**: JSON с динамическими полями из Bitrix24
- **last_synced**: Время последней синхронизации
- **sync_status**: Статус синхронизации

### FieldMapping (Маппинг полей)

- **id**: Уникальный идентификатор
- **entity_type**: Тип сущности (visit, clinic, doctor)
- **app_field_name**: Имя поля в приложении
- **bitrix_field_id**: ID поля в Bitrix24
- **field_type**: Тип поля (string, number, date, datetime, boolean, etc)
- **is_required**: Обязательное ли поле
- **display_name**: Отображаемое название поля
- **value_options**: JSON с возможными значениями для списочных полей
- **is_multiple**: Поддержка множественного выбора
- **entity_type_id**: ID типа сущности в Bitrix24
- **show_in_card**: Показывать ли поле на карточке

## API Endpoints

### Аутентификация

- `POST /auth/register` - Регистрация пользователя
- `POST /auth/login` - Авторизация пользователя
- `GET /auth/me` - Получение информации о текущем пользователе

### Визиты

- `GET /visits` - Получение списка визитов
- `POST /visits` - Создание нового визита
- `GET /visits/{visit_id}` - Получение информации о визите
- `PUT /visits/{visit_id}` - Обновление визита
- `PATCH /visits/{visit_id}/status` - Обновление статуса визита
- `DELETE /visits/{visit_id}` - Удаление визита

### Клиники

- `GET /clinics` - Получение списка клиник
- `POST /clinics` - Создание новой клиники
- `GET /clinics/{clinic_id}` - Получение информации о клинике
- `PUT /clinics/{clinic_id}` - Обновление клиники
- `DELETE /clinics/{clinic_id}` - Удаление клиники
- `GET /search/inn/{inn}` - Поиск клиники по ИНН

### Врачи

- `GET /doctors` - Получение списка врачей
- `POST /doctors` - Создание нового врача
- `GET /doctors/{doctor_id}` - Получение информации о враче
- `PUT /doctors/{doctor_id}` - Обновление врача
- `DELETE /doctors/{doctor_id}` - Удаление врача

### Синхронизация

- `POST /bitrix/sync/clinics` - Синхронизация клиник с Bitrix24
- `POST /bitrix/sync/visits` - Синхронизация визитов с Bitrix24
- `POST /bitrix/sync/doctors` - Синхронизация врачей с Bitrix24

### Маппинг полей

- `GET /admin/field-mappings` - Получение всех маппингов полей
- `GET /admin/field-mappings/{entity_type}` - Получение маппингов для конкретного типа сущности
- `POST /admin/field-mappings` - Создание нового маппинга
- `PUT /admin/field-mappings/{mapping_id}` - Обновление маппинга
- `DELETE /admin/field-mappings/{mapping_id}` - Удаление маппинга

## Интеграция с Bitrix24

### Настройка подключения

- API-эндпоинт: `https://crmwest.ru/rest/156/fnonb6nklg81kzy1/`
- Endpoint для webhook указывается в переменной окружения `BITRIX24_WEBHOOK_URL`

### Сущности Bitrix24

- **Визиты**: Smart Process 1054
- **Клиники**: Компании (Companies)
- **Врачи**: Контакты (Contacts) с типом DOCTOR

### Особенности синхронизации

- Двусторонняя синхронизация всех сущностей
- Поддержка динамических полей через таблицу `field_mappings`
- Статус синхронизации отслеживается через поля `last_synced` и `sync_status`

## Процесс аутентификации

1. Регистрация пользователя (только с email, который существует в Bitrix24)
2. Авторизация с выдачей JWT токена
3. Использование токена для всех защищенных API вызовов
4. Обновление токена через рефреш-токен (при необходимости)

## Известные проблемы и решения

### CORS и SSL

- Добавлен параметр `ssl=False` в запросах к Bitrix24 API для обхода проблем с сертификатами
- Настроены CORS заголовки для работы с разных доменов

### Синхронизация данных

- Для обхода проблем с дубликатами используется механизм отслеживания через `bitrix_id`
- При ошибках синхронизации поле `sync_status` устанавливается в 'error'

### Поля даты/времени

- Добавлена специальная обработка для полей типа "datetime" при синхронизации
- Используется формат ISO 8601 для хранения и передачи дат
- Корректное отображение и редактирование в компоненте DynamicFields

## Запуск в Production-режиме

1. Настройте переменные окружения в `.env` файле:

```
DATABASE_URL=postgresql://postgres:postgres@db:5432/visits
SECRET_KEY=your-secure-key-here
BITRIX24_WEBHOOK_URL=https://crmwest.ru/rest/156/your-webhook-key/
CORS_ORIGINS=http://yourdomain.com
```

2. Запустите приложение через Docker Compose:

```bash
docker-compose up -d --build
```

3. Проверьте работу приложения:
   - Frontend: http://yourdomain.com
   - Backend API: http://yourdomain.com/api
   - API Documentation: http://yourdomain.com/api/docs

## Рекомендации по разработке

1. Всегда используйте TypeScript для типизации на фронтенде
2. Добавляйте валидацию для всех входящих данных через Pydantic схемы
3. Следуйте принципу DRY (Don't Repeat Yourself)
4. Используйте React Query для кэширования данных и управления состоянием
5. Проверяйте совместимость с PWA принципами для оффлайн-работы
6. При добавлении новых полей в Bitrix24 обязательно создавайте соответствующие записи в таблице `field_mappings`
