# PRD: Изоляция архитектуры Beton CRM

## Цель проекта
Переработать архитектуру системы так, чтобы ВСЕ операции работали через локальную PostgreSQL базу данных. Интеграции (Bitrix24, Elasticsearch) становятся фоновыми синхронизаторами, которые наполняют и обновляют локальную БД.

## Принцип работы

### БЫЛО (текущая архитектура):
```
Форма → Elasticsearch → Bitrix24 API (fallback)
```
Проблема: Без Elasticsearch/Bitrix24 система не работает.

### СТАНЕТ (целевая архитектура):
```
                    ┌─────────────────────────────────┐
                    │     PostgreSQL (Single Source)  │
                    │  • Companies                    │
                    │  • Contacts                     │
                    │  • Nomenclature ✅              │
                    │  • Users, Forms, Submissions    │
                    └───────────────┬─────────────────┘
                                    │
            ┌───────────────────────┼───────────────────────┐
            │                       │                       │
            ▼                       ▼                       ▼
    [Frontend Forms]    [Bitrix24 Sync Job]    [Elasticsearch Indexer]
    (читает из БД)      (фоновая синхронизация) (опциональный индекс)
```

## Ключевой принцип
**Для пользователя НИЧЕГО не меняется** - те же формы, тот же поиск, та же скорость.
Меняется только внутренняя архитектура: все данные хранятся локально.

## Фазы реализации

### Фаза 1: Создание локальных таблиц контрагентов
Создать полноценные таблицы Companies и Contacts в PostgreSQL с:
- Всеми необходимыми полями (название, ИНН, телефон, email и т.д.)
- Full-text search через tsvector
- Полями для маппинга на Bitrix24 (bitrix_company_id, sync_status)
- Индексами для быстрого поиска

Задачи:
1.1. Создать entity Company.entity.ts с полями: id, name, shortName, inn, kpp, ogrn, legalAddress, actualAddress, phone, email, website, industry, companyType, notes, attributes, tags, isActive, bitrixCompanyId, syncStatus, lastSyncAt, syncError, searchVector
1.2. Создать entity Contact.entity.ts с полями: id, firstName, lastName, middleName, position, phone, email, companyId (FK), notes, attributes, tags, isActive, bitrixContactId, syncStatus, lastSyncAt, syncError, searchVector
1.3. Создать миграцию для таблиц companies и contacts с индексами GIN для searchVector, индексами на bitrix_id, inn
1.4. Создать CompanyRepository.ts с методами: findAll, findById, findByBitrixId, findByInn, search (fulltext), create, update, delete
1.5. Создать ContactRepository.ts с аналогичными методами
1.6. Создать CompanyService.ts с бизнес-логикой
1.7. Создать ContactService.ts с бизнес-логикой

### Фаза 2: API endpoints для контрагентов
Создать REST API для работы с компаниями и контактами.

Задачи:
2.1. Создать companyController.ts с методами: getAll, getById, search, create, update, delete
2.2. Создать contactController.ts с аналогичными методами
2.3. Создать companyRoutes.ts: GET /companies, GET /companies/:id, GET /companies/search, POST /companies, PUT /companies/:id, DELETE /companies/:id
2.4. Создать contactRoutes.ts с аналогичными роутами
2.5. Зарегистрировать роуты в app.ts

### Фаза 3: Переключение форм на локальную БД
Изменить frontend чтобы формы получали данные из локальной БД вместо Elasticsearch.

Задачи:
3.1. Создать client/src/services/companyService.ts с методами search, getById, getAll
3.2. Создать client/src/services/contactService.ts с аналогичными методами
3.3. Обновить useDynamicOptions.ts: для case 'companies' использовать CompanyService.search() как primary source
3.4. Обновить useDynamicOptions.ts: для case 'contacts' использовать ContactService.search() как primary source
3.5. Проверить что формы работают корректно с новыми источниками данных

### Фаза 4: Фоновая синхронизация с Bitrix24
Создать систему фоновой синхронизации данных между локальной БД и Bitrix24.

Задачи:
4.1. Создать Bitrix24CompanySyncService.ts с методами: importAll, importById, importBatch, exportTobitrix
4.2. Создать Bitrix24ContactSyncService.ts с аналогичными методами
4.3. Создать Bitrix24NomenclatureSyncService.ts для синхронизации товаров
4.4. Создать SyncScheduler.ts для запуска синхронизации по расписанию (cron)
4.5. Создать API endpoints для ручного запуска синхронизации: POST /api/sync/companies, POST /api/sync/contacts, POST /api/sync/nomenclature
4.6. Добавить логирование и отслеживание статуса синхронизации

### Фаза 5: Обработка создания сделок
При создании заявки (submission) - создавать сделку в Bitrix24 асинхронно.

Задачи:
5.1. Изменить SubmissionService: сначала создавать заявку в локальной БД, потом асинхронно отправлять в Bitrix24
5.2. Добавить очередь задач (или использовать существующую) для отправки в Bitrix24
5.3. Обрабатывать ошибки синхронизации с Bitrix24 (retry, logging)
5.4. Обновлять bitrixDealId в submission после успешного создания сделки

### Фаза 6: Admin Panel для контрагентов
Создать интерфейс управления компаниями и контактами в админке.

Задачи:
6.1. Создать компонент CompanyList.tsx со списком компаний, поиском и фильтрами
6.2. Создать компонент CompanyForm.tsx для создания/редактирования компании
6.3. Создать компонент ContactList.tsx со списком контактов
6.4. Создать компонент ContactForm.tsx для создания/редактирования контакта
6.5. Создать компонент SyncStatus.tsx для отображения статуса синхронизации
6.6. Добавить страницы в админ-панель: /admin/companies, /admin/contacts
6.7. Добавить возможность импорта из Excel

### Фаза 7: Опциональный Elasticsearch
Сделать Elasticsearch опциональным - для расширенного поиска, если включен.

Задачи:
7.1. Добавить переменную окружения ELASTICSEARCH_ENABLED
7.2. Создать ElasticsearchIndexer.ts который индексирует данные из PostgreSQL в Elasticsearch
7.3. Изменить searchController.ts: использовать Elasticsearch только если ELASTICSEARCH_ENABLED=true
7.4. При отключенном Elasticsearch все поиски идут через PostgreSQL fulltext search
7.5. Добавить в docker-compose profiles для опционального запуска Elasticsearch

## Технические требования

### База данных
- PostgreSQL 16
- Full-text search с tsvector и GIN индексами
- Русская морфология для поиска

### API
- REST endpoints для CRUD операций
- Поиск через GET параметры: ?query=...&limit=20
- Пагинация для списков

### Синхронизация
- Фоновые jobs для синхронизации с Bitrix24
- Инкрементальная синхронизация (только изменения)
- Обработка конфликтов (local wins / remote wins / merge)
- Логирование всех операций синхронизации

### Совместимость
- API должен быть обратно совместим
- Frontend не должен требовать изменений (кроме использования новых сервисов)
- Существующие данные должны быть сохранены

## Критерии успеха
1. Система работает без Elasticsearch (можно полностью отключить)
2. Система работает без доступа к Bitrix24 API (offline mode)
3. Поиск компаний/контактов работает через PostgreSQL с временем отклика <200ms
4. Синхронизация с Bitrix24 происходит в фоне и не блокирует пользователя
5. Для пользователя интерфейс остается без изменений
